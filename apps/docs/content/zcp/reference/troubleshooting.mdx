Diagnose common Zerops Control Plane (ZCP) problems with setup, tokens, VPN, deploys, logs, public URLs, and workflow state. This page is symptom-driven — find the symptom that matches what you are seeing and follow the concrete next check. For a category-driven view of deploy failures (`build`, `start`, `verify`, `network`, `config`, `credential`, `other`), see [Failure categories](/zcp/reference/failure-categories).

:::caution Public preview
Zerops Control Plane (ZCP) is in public preview. Use it for development and staging projects, keep production in a separate Zerops project, and expect setup details to change.
:::

## Start with status

Before anything else, ask the agent to read ZCP status. Most "the agent forgot," "ZCP cannot see anything," and "I lost the thread" symptoms are recovered in one prompt:

```text
Read ZCP status and tell me where this project stands.
```

Status is the universal first move. It rebuilds the agent's view from live platform state — services, their state, recent deploys, recent failures — without you having to re-explain the project. If the answer comes back coherent, the session is fine and you can carry on. If the answer reveals a token, VPN, or platform problem, the rest of this page maps the symptom to the fix.

The same prompt is the right recovery move when:

- A long Claude Code session has lost the thread of where the work was.
- You closed the browser tab or restarted the editor mid-task.
- The agent answers vaguely about services or asks you to repeat what the project contains.
- You are not sure whether the last deploy actually succeeded.

If status itself fails — the agent reports "cannot reach ZCP" or returns a token error — the problem is below the workflow layer. Continue to the relevant section.

## Token errors

Token rejection happens at startup. ZCP refuses to come up with a token that has the wrong shape, and the error names the specific reason.

| Symptom | Likely cause | Next move |
|---|---|---|
| `Token accesses N projects; use project-scoped token` | A multi-project (or full-access) token was supplied. | Generate a project-scoped token in the Zerops dashboard and replace the value. See [Tokens and project access](/zcp/security/tokens-and-project-access). |
| `Token has no project access` | The token authenticates a user but cannot reach any project. | Grant the token project access (or generate a new project-scoped token) and replace the value. |
| `No authentication found: set ZCP_API_KEY or log in with zcli` | No token is reaching ZCP. | Add `ZCP_API_KEY` under the `env` block of `.mcp.json`, or run `zcli login <token>`. |
| `cli.data found but token is empty` | zCLI has a config file but no token in it. | Run `zcli login <token>` with a project-scoped token. |
| API replies with 401 / `AUTH_TOKEN_EXPIRED` | The token was revoked or has expired. | Generate a new project-scoped token, replace the value, and restart the agent. |

After replacing the token in `.mcp.json`, restart Claude Code. ZCP reads `ZCP_API_KEY` once at process startup; the live session still holds the old value in memory.

```bash
# After editing .mcp.json, relaunch Claude Code from the project root
cd path/to/your/project
# launch your Claude Code client; it picks up the new env on startup
```

If the agent still reports the same rejection after restart, the value in the file is still the wrong shape. Generate a fresh project-scoped token and try again.

## VPN errors (local install)

The local agent bridge reaches Zerops managed services over the project [VPN](/references/networking/vpn). The hosted workspace does not need a VPN — it is already on the project's private network. These symptoms only apply to the local bridge.

| Symptom | Likely cause | Next move |
|---|---|---|
| `connection refused` against a managed service hostname (e.g. `db`, `redis`) | VPN is not up, or it dropped after sleep / network change. | `zcli vpn up <project-id>` — bring the tunnel back. |
| `no route to host` for `db.zerops` or similar | Same as above; the search domain only resolves through the VPN. | `zcli vpn up <project-id>`. |
| Local app starts but cannot connect to Postgres / Valkey / storage | VPN is up but the `.env` is stale, or VPN is down. | First check `nc -zv db 5432 -w 3` (or the right port). If that fails, restart the VPN. If it succeeds, regenerate the `.env`. |
| `zcli vpn up` prompts repeatedly for sudo | VPN setup needs root each time the tunnel comes up. | This is expected on Linux and macOS — VPN setup requires admin/root. |

A representative recovery loop:

```bash
# 1. Confirm the VPN is up
nc -zv db 5432 -w 3

# 2. If it is not, bring it back
zcli vpn up <project-id>

# 3. After VPN is healthy, regenerate stale env values
# (ask Claude Code to do this; it reads live values from the project)
```

VPN drops with sleep, network change, and idle timeouts. Treat re-running `zcli vpn up` as a routine recovery, not a setup error.

## Deploy errors

A failed deploy carries a structured failure classification. The agent reads that first; you can route to the matching category if you want to read it yourself. The full classification reference is at [Failure categories](/zcp/reference/failure-categories).

| Symptom | Likely category | Next move |
|---|---|---|
| Build container exits non-zero — "command not found", missing module, npm 404 | `build` | Read `buildCommands` in `zerops.yaml`. Add missing packages or fix typos. |
| Container OOM-killed during build | `build` | Increase build container RAM (`build.containerOptions` in `zerops.yaml`) or split heavy install steps. |
| `prepareCommands` fails — package not found, permission denied | `start` (prepare runs before runtime is fully up) | Check the package name for the runtime base distro (Alpine/Debian); prefix privileged ops with `sudo`. |
| Runtime crashes immediately — `EADDRINUSE`, `Cannot find module`, missing env var | `start` | Read runtime logs scoped to the failed deploy. The classification names the likely cause. |
| Database connection refused during init | `start` | Confirm the DB service is `RUNNING` (`zerops_discover`); confirm env var refs (`${db_*}`) resolve. |
| `signal: killed` over SSH during deploy | `network` | Source container OOM-killed. Scale source RAM up and retry. |
| `Authentication failed` on git push | `credential` | Check `GIT_TOKEN` (hosted) or your local git credential helper (local). |
| `Cannot find corresponding setup` from zcli | `config` | `zerops.yaml` has multiple setup blocks; pass the right `setup` name on deploy. |
| Preflight refuses with `INVALID_ZEROPS_YML` | `config` | Read the field-level reasons in `apiMeta` — fix the rejected fields and retry. |

For any of these, the diagnostic next step is the same — ask the agent to read the deploy events, build logs, and runtime logs:

```text
Show me the build logs for the last failed deploy of appdev, and read the failure classification.
```

Build logs and runtime logs are different streams. A build failure does not show in runtime logs, and a runtime crash does not show in build logs. If the agent pastes the wrong stream, the diagnosis will miss.

## Verify errors

A deploy can succeed (build green, runtime running) and still fail to verify. Verify is a separate gate — reachability plus the requested behavior. Some common shapes:

| Symptom | Likely cause | Next move |
|---|---|---|
| Service status is `RUNNING` but the route returns 500 / 404 | The app started but the requested behavior is broken. | Read runtime logs at request time; the agent confirms whether the route is wired up and the dependencies it needs. |
| HTTP probe on the public URL returns a connection error | Subdomain access is still propagating, or the runtime is not bound to `0.0.0.0`. | Wait 30-60 seconds and retry. If it stays broken, see the next section. |
| Service status is `READY_TO_DEPLOY` after a deploy | Build succeeded but the runtime never started (start command failed). | Read runtime logs scoped to `ContainerCreationStart` — the start command itself is the cause. |
| `RUNNING` but log output shows continuous restart | The start command exits and is being restarted by the platform. | Read runtime logs over the last few minutes; the exit reason is in the logs. |

Verify failures route through the same failure classification as deploy failures. The agent reads that classification, decides whether to fix and retry or escalate.

## Public URL errors

Public access is auto-enabled on the first deploy for eligible services. If the URL still does not work after deploy, work through the symptoms in order — the platform may still be propagating, or the service may not be eligible for public access.

| Symptom | Likely cause | Next move |
|---|---|---|
| URL returns "Not Found" or empty response right after first deploy | Subdomain is still propagating. | Wait 30-60 seconds and re-fetch. The agent's verify step retries automatically. |
| Subdomain stays disabled even after a successful deploy | The service is a worker (no HTTP) or a non-eligible mode. | Workers have no public URL by design. Confirm the service is meant to serve HTTP. |
| 502 Bad Gateway persists after deploy + wait | The runtime is not bound to `0.0.0.0`, or the port in `zerops.yaml` does not match the bound port. | Read `zerops.yaml` `run.ports[]`. Confirm the start command binds the same port on `0.0.0.0`, not `127.0.0.1`. |
| Custom domain not resolving | DNS propagation, or the domain is not configured on the service. | See [Public access](/guides/public-access) for the full domain setup. |

If the service really should have a public URL and auto-enable did not catch it, you can ask the agent to enable subdomain access explicitly:

```text
Enable subdomain access on appdev and verify the URL responds.
```

The agent runs the equivalent operation, waits for HTTP-readiness, and reports the URL.

## Local install errors (`zcp init`, `.mcp.json`, `.env`)

Local-bridge setup has a few moving pieces — the binary, the generated config, the token, and the `.env`. The most common stumbles:

| Symptom | Likely cause | Next move |
|---|---|---|
| Re-running `zcp init` made the agent stop working | `zcp init` rewrites `.mcp.json` from the template; your `ZCP_API_KEY` is gone. | Re-add the `env` block with `ZCP_API_KEY` and restart Claude Code. See [Tokens and project access](/zcp/security/tokens-and-project-access). |
| Claude Code does not list the `zerops` MCP server | Claude Code launched from the wrong directory. | Quit and relaunch from the project root that contains `.mcp.json`. |
| Agent says it cannot find a `zerops_*` tool | Claude Code's MCP introspection cached an older tool list. | Restart Claude Code so it re-reads the MCP server and picks up the current tool catalog. |
| Local app reads stale env values | `.env` is a snapshot taken at generation time. | Regenerate after any project env-variable change — ask the agent to re-run the env-file generator. |
| `zcp` not found on `PATH` after install | The install script wrote to `~/.local/bin` and that is not on `PATH`. | Add `~/.local/bin` to `PATH` in your shell rc and reload. |

Re-running `zcp init` is safe for everything except `.mcp.json` — `CLAUDE.md` preserves your edits outside the managed markers, but `.mcp.json` is rewritten clean. If you re-init, treat re-adding the token as routine.

## Workflow state errors

Workflow state errors usually surface as "the agent forgot," "no project found," or "session lost" symptoms. Almost all of them recover with a status read.

| Symptom | Likely cause | Next move |
|---|---|---|
| Agent says "no project found" or "I have no context" mid-conversation | The agent's chat memory drifted from real platform state. | Ask for status: `Read ZCP status and tell me where this project stands.` Status reads live state and rebuilds the picture. |
| Agent asks you to re-describe the project from scratch | Same as above — the chat lost the thread but the platform did not. | Same fix. Status returns the project's services and recent activity. |
| Agent loops on the same failed deploy without progressing | Five attempts without progress is a stop signal — see [Build and verify an app](/zcp/workflows/build-and-verify-app). | Stop the loop and ask the agent to summarize what it has tried and what evidence is still missing. |
| Agent picks the wrong service when you said "the runtime" | In `standard` shape there are two runtimes (dev + stage); ambiguous prompts can land on the wrong one. | Name the hostname explicitly: `Deploy appdev` rather than `deploy the runtime`. |
| A mutation tool returned a terse response without lifecycle context | Mutation responses are intentionally terse; lifecycle context is recovered by reading status. | Ask for status if you want the full picture. |
| `WORKFLOW_REQUIRED` or `WORKFLOW_ACTIVE` from a tool | The tool needs a workflow open (or refuses because one is already open). | The error message names which workflow to start or close. Follow the named recovery. |

Status is not just for cold starts. Any time the conversation drifts from the project, asking ZCP for current state realigns the agent with real state in one prompt.

## Gotchas

1. **Status is the first move, not the last.** When something is unclear — "did that deploy succeed?", "what state is the project in?" — ask for status before debugging anything else. Most "the agent is confused" cases recover from a status read in one turn.
2. **Restart Claude Code to refresh the MCP tool list.** Claude Code reads the `zerops_*` tool catalog at startup. If ZCP's tool surface changes (new install, version update) and the agent says it cannot find a tool, quit and relaunch the editor so it re-introspects the MCP server.
3. **Build logs and runtime logs are different streams.** A build failure shows in build-container logs; a runtime crash shows in runtime-container logs. Asking for the wrong one returns nothing useful — make sure the agent is reading the stream that matches the phase that failed.
4. **VPN drops are routine, not setup errors.** Sleep, network change, and idle timeouts will drop the VPN tunnel. `zcli vpn up <project-id>` brings it back. The error message ("connection refused" / "no route to host") looks like an app bug but is almost always a network move.
5. **Re-running `zcp init` does not preserve the token in `.mcp.json`.** The regenerated file ships without `ZCP_API_KEY`. Re-adding the `env` block is part of the re-init flow, not a separate setup step.
