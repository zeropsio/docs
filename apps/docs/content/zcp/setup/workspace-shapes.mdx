Zerops Control Plane (ZCP) describes the workspace by two questions: where the agent's files live, and whether the project has a separate stage runtime to promote to. The five names you may see in agent responses (`standard`, `dev`, `simple`, `local-stage`, `local-only`) are the combinations of those two answers — each one is the agent telling you which combination it picked up from your project.

You almost never pick a shape directly. The agent reads it from the project state and explains the choice. This page is for understanding what it picked and why.

## Where do the agent's files live?

| Files live | Workspace | What that means |
|---|---|---|
| In the Zerops project, on runtime service filesystems | Hosted workspace | The agent edits and deploys from inside the project. Filesystem access via SSHFS mounts. See [Provision a hosted workspace](/zcp/setup/hosted-workspace). |
| On your laptop, in your local checkout | Local agent bridge | The agent edits your local files and deploys from your laptop to the project. See [Use ZCP from your laptop](/zcp/setup/local-agent-bridge). |

This is the same hosted-vs-local question covered in [Choose your workspace](/zcp/setup/choose-workspace). It also sets the security profile — see [Trust model](/zcp/security/trust-model).

## Is there a separate stage runtime?

| Stage situation | When to use | Typical names you'll see |
|---|---|---|
| Dev runtime plus an explicit stage runtime | You want a separate target the agent verifies before promoting work | `appdev` + `appstage`. Agent calls this `standard`. |
| One mutable dev runtime, no stage yet | You want a single iteration target without splitting dev from stage | `appdev` (or similar). Agent calls this `dev`. |
| One runtime, no dev/stage split | Small apps, demos, static sites | `app` (or similar). Agent calls this `simple`. |
| Local checkout with one Zerops runtime as the stage target | You develop locally with hot reload and deploy to a Zerops stage runtime | The local checkout + a linked Zerops runtime. Agent calls this `local-stage`. |
| Local checkout, no Zerops runtime linked yet | The project is local-only or has only managed services | The local checkout. Agent calls this `local-only`. |

Stage is a deploy target for dev and review, not production. Production lives in a separate Zerops project (see [Production policy](/zcp/security/production-policy)).

The stage hostname is **explicit**. The agent does not invent it from the dev hostname — `appstage` next to `appdev` is one example, not a rule. The confirmed project state or setup plan supplies the stage hostname, and the agent uses what is named there.

## Workspace topology is not service scaling mode

Workspace shape (`standard`, `dev`, `simple`, `local-stage`, `local-only`) is ZCP's name for the project topology. Zerops service `mode` (`HA`, `NON_HA`) is the scaling shape of an individual managed service — see [Scaling](/features/scaling). The two are independent: a `standard` workspace can use `NON_HA` services, a `simple` workspace can include managed services configured as `HA`.
