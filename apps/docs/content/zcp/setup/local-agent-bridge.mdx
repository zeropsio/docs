title: "Use ZCP locally"
description: "Run ZCP locally so your editor and agent can discover Zerops services, generate credentials, and deploy to Zerops."
---

Run Zerops Control Plane (ZCP) as a binary on your laptop so your editor and agent can operate one Zerops project. With `zcli vpn up <project-id>`, your laptop joins the project's private network — the same network your services use. ZCP is the bridge between your local editor, the agent, and the project.

:::caution Public preview
Zerops Control Plane (ZCP) is in public preview. Use it for development and staging projects, keep production in a separate Zerops project, and expect setup details to change.
:::

The local agent bridge is for developers who already have a comfortable local setup — your editor, your shell, your dev server — and want a coding agent that can drive Zerops without giving up your machine. If you are evaluating ZCP for the first time, the [hosted workspace](/zcp/setup/hosted-workspace) is a faster start.

You don't need ZCP to develop locally against a Zerops project. `zcli vpn up <project-id>` is enough to reach managed services by hostname from your laptop and run your dev process against them — see [Local Development with Zerops](/guides/local-development) for the platform-side picture. ZCP is what you add when you also want a coding agent operating the project from the same machine: discovering services, generating credentials, deploying, reading logs, verifying.

:::caution Supervise the agent client
The local bridge runs the agent on your dev machine. ZCP itself stays project-scoped, but the agent process inherits your user — what your client (Claude Code, etc.) does with file edits, shell commands, and other tools is decided by your client's permissions, not ZCP's. Set those permissions accordingly. See [Trust model](/zcp/security/trust-model) for the full picture.
:::

## Prerequisites

- A Zerops project. Create one from the [Zerops dashboard](https://app.zerops.io/dashboard/project-add) or from a [recipe](https://app.zerops.io/recipes).
- [zCLI](/references/cli) installed on your machine and authenticated.
- A **project-scoped Zerops token** (steps below). Multi-project tokens are refused.
- A **Claude subscription** (Pro or Max) and Claude Code installed locally. Claude Code handles login through its own flow; ZCP does not see your subscription credentials.

## Get a project-scoped Zerops token

ZCP needs a Zerops API token scoped to **one project** — multi-project and full-access tokens are refused at startup. Generate one in the Zerops dashboard, then keep the value handy for the **Add your token** step below. Full walkthrough, rejection rules, and rotation: [Tokens and credentials](/zcp/security/tokens-and-project-access).

## Install ZCP

Install the `zcp` binary on your machine:

```bash
curl -sSfL https://raw.githubusercontent.com/zeropsio/zcp/main/install.sh | sh
```

The script downloads the latest release for your platform into `~/.local/bin` (or `/usr/local/bin` if run as root). Verify with `zcp version`. If `zcp` is not on your `PATH`, make sure `~/.local/bin` is in `PATH` and reload your shell.

## Initialize ZCP in your project directory

Open a terminal in your project's working directory (the one that contains your code) and run:

```bash
zcp init
```

`zcp init` writes:

- `.mcp.json` — the MCP configuration Claude Code reads to discover ZCP
- `CLAUDE.md` — instructions for the agent about how to operate ZCP in this project
- `.claude/settings.local.json` — Claude Code per-project settings
- `~/.config/zerops/aliases` plus a shell RC sourcing line — today this defines one `zcl` alias that launches Claude Code

ZCP creates `.zcp/state/` later, on the first state write for adoption, sessions, or workflow progress. Re-running `zcp init` refreshes the generated files. `CLAUDE.md` preserves your edits outside the managed markers, but `.mcp.json` is rewritten from the template — if you added `ZCP_API_KEY` to it, check the file afterward and re-add the token. The regenerated `.mcp.json` ships without a token.

## Add your token

`zcp init` writes a token-less `.mcp.json`. Add your project-scoped token under an `env` block (`ZCP_API_KEY`), gitignore the file, and launch Claude Code from your project root so it picks up the config. The exact JSON shape and rotation rules live in [Tokens and credentials → Where the token lives](/zcp/security/tokens-and-project-access#where-the-token-lives--hosted-vs-local).

Claude Code starts and lists the MCP servers it found. The `zerops` server (that's the ZCP MCP server) should be in the list.

### Sanity check

Once Claude Code is running, verify the connection:

```text
Use ZCP to list the services in this project.
```

A working connection answers with the project's runtime and managed services and their state. If the agent says it cannot reach ZCP, see [Troubleshooting](/zcp/reference/troubleshooting) — the most common issues are token-scope mismatches and starting Claude Code from the wrong directory.

## Connect the project VPN

ZCP reaches managed services (databases, caches, storage) through the Zerops project [VPN](/references/networking/vpn). Bring it up with zCLI:

```bash
zcli vpn up <project-id>
```

You will be prompted for sudo or admin — VPN setup needs root on Linux and macOS. ZCP cannot start the VPN for you; this is your one manual step.

After the VPN is up, you can resolve project services by hostname (`db`, `cache`, etc.) from your laptop.

VPN gives you network reach. It does not expose env values. ZCP generates those for you in the next step.

## Generate a local `.env`

Ask Claude Code to generate the env file from your project:

```text
Use ZCP to generate a .env file for my local app.
```

ZCP reads `run.envVariables` for the requested service from your local `zerops.yaml`, resolves env-template references (the `${hostname_var}` syntax used in `zerops.yaml`) from the platform, and writes a `.env` in your working directory. Cross-service references in compound values resolve recursively — for example `DATABASE_URL=postgres://${db_user}:${db_password}@${db_hostname}:${db_port}/${db_dbName}` lands as a fully-resolved connection string.

The tool call must include the target service hostname, your working directory must contain `zerops.yaml`, and that file must have a matching `setup:` entry with non-empty `run.envVariables`. Without those, generate-dotenv returns an error instead of guessing. Your local app reads the same hostnames and credentials it would see when deployed to Zerops, reached over VPN instead of from inside the project's private network.

The `.env` is a snapshot, not a live link. Regenerate after changing project env variables.

## Link a deploy target

Local mode needs one Zerops runtime linked as your stage target so the agent has somewhere to deploy from your working directory. If your project has exactly one runtime, ZCP picks it up automatically. If it has multiple, the agent asks which one to link — answer by hostname (for example, `appstage`).

If no runtime is linked yet (the project has only managed services, or you haven't decided), the agent's default deploy path refuses with a clear hint. You can still develop locally against the managed services, and git-push delivery can still work if your project is configured for it — see [Choose how finished work ships](/zcp/workflows/delivery-handoff). Ask the agent to link a runtime by hostname when you're ready.

The shape names the agent uses (`local-stage` once linked, `local-only` until then) and the full hosted-vs-local taxonomy live in [Choose your workspace](/zcp/setup/choose-workspace#workspace-shapes).

## What stays your tool's job

ZCP does not run your local dev server. Whether you use Vite, Valet, Docker Compose, your IDE's runner, or something else, the dev process stays your tool's responsibility. ZCP works alongside it.

ZCP also does not mount Zerops runtime filesystems on your laptop. The hosted workspace gives the agent SSHFS access to runtime services; the local bridge does not. If you need to read runtime files from your machine, use [SSH](/references/networking/ssh) directly.

For the full hosted-vs-local comparison, see [Choose your workspace](/zcp/setup/choose-workspace).

## Gotchas

1. **VPN goes down with sleep.** When your laptop sleeps or your network changes, the VPN often drops. Bring it back with `zcli vpn up <project-id>` before the agent retries.
2. **Each project directory needs its own `zcp init`.** State and tokens are per-project. Switching projects on your laptop means switching directories — Claude Code picks up the wrong ZCP connection if you launch from the wrong root.
3. **Multi-project tokens are rejected at startup.** A token granting access to multiple projects (or no project) is refused. Generate a per-project token (steps above).
4. **The `zerops` server name is fixed.** `.mcp.json` keys the ZCP MCP server as `zerops`. Don't rename it; ZCP and Claude Code expect that key.
5. **The `.env` is a snapshot.** Regenerate after changing project env variables, not before.
6. **Production stays out of this loop.** The local bridge is for development and staging projects. See [Production boundary](/zcp/security/production-policy).
