title: "What ZCP lets agents do"

---

Zerops Control Plane (ZCP) groups platform operations by user job: discover, deploy, verify, operate, recover, and hand off. You ask for an outcome in plain language; the agent picks the right operations and reports what changed.

## Discover

The agent reads the project before changing anything: which services exist, what state they are in, what env variables they expose, what the recent build and deploy timeline looks like. When you ask "what is in this project?", you do not describe it — the agent asks ZCP and tells you. The same surface answers questions about runtime activity and recent failures.

## Deploy

The agent ships code through the standard Zerops [build and deploy pipeline](/guides/deployment-lifecycle). The deploy call blocks until the build finishes and the runtime is ready — the agent waits with you, not for you. When several services need to ship together, the agent coordinates the deploys and reports each result.

Deploy success is not the same as verify success. A deploy that finished is a deploy that built and started; whether the app actually does what you asked is a separate check.

## Verify

After a deploy, the agent runs post-deploy health checks: service status, error logs, startup detection, an HTTP probe on the public URL. If reachability passes, the agent then checks the requested behavior — opening the URL, hitting an endpoint, inspecting a database row.

If verify fails, the agent reads a structured failure classification before raw logs and routes to the right fix. See [Failure categories](/zcp/reference/failure-categories).

## Operate

For service lifecycle — start, stop, restart, reload, [scaling](/features/scaling), [public access](/references/networking/public-access) — the agent uses scoped operations that read and change one project at a time. [Environment variables](/features/env-variables) are read and set the same way, at service or project scope, with preprocessor support so the agent can generate strong secrets in place.

When ZCP runs in the hosted workspace, the agent has filesystem access to other services through SSHFS mounts and can run a long-lived dev process inside another container.

## Recover

Every failed deploy carries a structured failure classification: a category, a likely cause, a suggested next action. The agent reads this before the raw logs. See [Failure categories](/zcp/reference/failure-categories).

If the agent loses context — a long session, a browser crash, a closed tab — it asks ZCP for status and resumes from current platform state. The recovery does not depend on the prior chat. See [How ZCP works](/zcp/concept/how-it-works).

## Hand off

When the work is done, you and the agent decide how the next change ships: keep deploying directly, commit and push to git, or hand off to your CI. ZCP records the choice and configures any push credentials and build integrations needed. See [Choose how finished work ships](/zcp/workflows/delivery-handoff).

For a different kind of handoff — turning a deployed service into reusable project files — see [Package a running service for reuse](/zcp/workflows/package-running-service).

## What needs your explicit confirmation

Service deletion and wholesale service replacement (override import) sit behind a confirmation gate: the agent cannot delete a service or replace a service with failed deploy history without an explicit yes from you in the same conversation. The agent reads the failure first, then asks. See [Destructive actions and confirmations](/zcp/security/destructive-actions).

## What's not here

A few capabilities are intentionally out of scope:

- **Authoring new recipes.** Recipes are pre-built starters Zerops publishes; ZCP users select them.
- **Your local dev server in local mode.** ZCP does not start, stop, or watch the dev process you run on your own machine. That stays your tool's job.
- **Production operations.** Keep production in a separate Zerops project and operate it through Zerops UI, the API, or your CI. See [Production policy](/zcp/security/production-policy). Production data restore is owned by [Backup](/features/backup), not ZCP.
