Project-scoped Zerops tokens, git credentials, and the confirmation gates Zerops Control Plane (ZCP) enforces before destructive actions. The token shape ZCP accepts, where it lives, how it relates to git credentials, and what the agent cannot do without your explicit consent — these are the practical edges of project boundary. For the full hosted-vs-local boundary picture, see [Trust model](/zcp/security/trust-model).

## Required token shape — project-scoped Zerops API token

ZCP needs a Zerops API token that resolves to **exactly one project** at startup. In the Zerops dashboard, the integration-token type that produces this is **Custom access per project** scoped to a single project — that is the recommended (and effectively required) shape. The other two integration-token shapes (*Full access to all projects* and *Read access to all projects*) typically resolve to more than one project for any non-trivial account; ZCP refuses to start whenever the resolved project count is anything other than one. See [Roles & Permissions — Integration Tokens](/features/rbac#integration-tokens) for how Zerops scopes the token at the platform layer. The token authorizes ZCP to read services, env vars, and recent activity in that one project, and to drive deploys, logs, and verification against it.

Generate one:

1. Open [Settings → Access Tokens Management](https://app.zerops.io/settings/token-management) in the Zerops dashboard.
2. Click **Create Token** and give it a name (for example `zcp-<project-slug>`).
3. Pick **Custom access per project** as the access level.
4. Add the project the agent will operate against, set its permission to **Full access**, and create the token.
5. Copy the value once. Zerops shows it only at creation time; if you lose it you have to regenerate it.

The token's blast radius equals the project. Anything outside the project — other projects in the same organization, account-level settings, billing — stays out of reach. See [Roles & Permissions](/features/rbac) for how Zerops scopes access tokens at the platform layer.

## Rejected token shapes — multi-project, no-project, expired

ZCP validates the token at startup and refuses to come up with the wrong shape. Three rejection modes matter:

| Token shape | What ZCP does | Why |
|---|---|---|
| Multi-project (e.g. full-access) | Refuses with `TOKEN_MULTI_PROJECT` | One ZCP connection equals one project; ZCP will not pick which one to drive. |
| No project access (zero-project) | Refuses with `TOKEN_NO_PROJECT` | The token authenticates a user but cannot reach any project. There is nothing for ZCP to operate on. |
| Read-only token | Fails on the first mutation | A read-only token passes startup auth but Zerops rejects the first deploy / env write / build call with a permission error. ZCP relies on Zerops RBAC for capability enforcement. |
| Expired or revoked | Refuses with `AUTH_TOKEN_EXPIRED` or `AUTH_REQUIRED` | The token no longer authenticates against the Zerops API. |

A typical multi-project rejection looks like this in the agent's view:

```text
Token accesses 4 projects; use project-scoped token
Recovery: Create a project-scoped token in Zerops GUI or set project via zcli scope
```

The fix is always the same: generate a per-project token and replace the value (or, if you are running ZCP through `zcli`'s credentials, switch the active scope to the target project before starting ZCP).

## Where the token lives — hosted vs local

ZCP reads the token from environment at startup. It does not write the token anywhere else, and it does not exchange it for a derived credential. Where the env var comes from depends on whether ZCP runs hosted or local:

| Workspace | Where `ZCP_API_KEY` comes from | Who provisions it |
|---|---|---|
| Hosted workspace (ZCP service inside the project) | The `zcp` service container env | Zerops platform — injected automatically when the workspace boots. You do not paste the token by hand. |
| [Local agent bridge](/zcp/setup/local-agent-bridge) | The `env` block of `.mcp.json` in the project directory | You — added manually after `zcp init`. |

In the hosted workspace, opening the workspace from the Zerops dashboard is enough. The container starts with `ZCP_API_KEY` already populated, and Claude Code launches against it without asking for a credential.

In the local agent bridge, `zcp init` writes a token-less `.mcp.json` template:

```json
{
  "mcpServers": {
    "zerops": {
      "command": "zcp",
      "args": ["serve"]
    }
  }
}
```

You add the project-scoped token under an `env` block:

```json
{
  "mcpServers": {
    "zerops": {
      "command": "zcp",
      "args": ["serve"],
      "env": {
        "ZCP_API_KEY": "<your-project-scoped-token>"
      }
    }
  }
}
```

Gitignore `.mcp.json` so the token does not leave the machine. Each project directory has its own `.mcp.json` and its own token — switching projects on your laptop means switching directories, not editing one shared file. See [Use ZCP locally](/zcp/setup/local-agent-bridge) for the full local setup walkthrough.

## Git credentials — separate from `ZCP_API_KEY`

Three names, three jobs — keeping them straight prevents most of the credential confusion in [delivery handoff](/zcp/workflows/delivery-handoff):

| Name | What it authorizes | Where it lives |
|---|---|---|
| `ZCP_API_KEY` | ZCP itself, against the Zerops API | Container env (hosted) or `.mcp.json` env block (local) |
| `GIT_TOKEN` | A git push from a ZCP-managed environment to a remote (GitHub, GitLab) | Hosted: project env var on the ZCP service. Local: not applicable — the user's git credential helper handles it. |
| `ZEROPS_TOKEN` | A GitHub Actions workflow running `zcli` against Zerops | A repository secret in GitHub Actions |

`GIT_TOKEN` is a separate concern from `ZCP_API_KEY`. It only matters when delivery uses the **git-push** close-mode and ZCP is the one performing the push (the hosted workspace path). In the local bridge, the developer's own git CLI talks to the remote — SSH keys or credential manager already handle authentication, and ZCP does not need a credential of its own. If `GIT_TOKEN` is present in the hosted workspace's project env when you run **git-push setup**, ZCP reuses it for the push instead of asking for a new credential.

`ZEROPS_TOKEN` is the GitHub Actions secret used by the `actions` build integration. It is a **Zerops API token, not a GitHub token** — the workflow uses it to authenticate `zcli` against Zerops. It can be the same project-scoped token already behind `ZCP_API_KEY` (one credential, one rotation surface) or a separate Zerops token if your team prefers credential separation. The agent prepares the `gh secret set` command for you, with the value pulled from the right place for your environment:

```bash
# Hosted workspace — read ZCP_API_KEY from the container env
gh secret set ZEROPS_TOKEN -b "$ZCP_API_KEY" -R <owner>/<repo>

# Local install — extract ZCP_API_KEY from .mcp.json
gh secret set ZEROPS_TOKEN -b "$(jq -r '.mcpServers.zerops.env.ZCP_API_KEY' .mcp.json)" -R <owner>/<repo>
```

Run the command in a terminal authenticated to the GitHub repository (`gh auth login` first if needed). The literal token value never crosses the MCP wire — the shell expands `$ZCP_API_KEY` or `jq` reads `.mcp.json` at command time.

## Rotation

When a token is rotated in the Zerops dashboard, the new value replaces the old one at the source:

- **Hosted workspace** — the project env value is updated; ZCP picks up the new value the next time the workspace boots (or the `zcp` service restarts).
- **Local agent bridge** — paste the new token into the `env` block of `.mcp.json`, then restart Claude Code so the new ZCP process inherits the updated env.
- **GitHub Actions secret** — `gh secret set ZEROPS_TOKEN ...` (or the equivalent in the GitHub UI) replaces the secret. The next workflow run uses the new value.

Rotations on `ZCP_API_KEY` and `ZEROPS_TOKEN` are independent unless you reused the same Zerops token for both. If you did, rotate it in the dashboard once, then update both surfaces.

## Confirmation gates for destructive actions

A valid token does not unlock everything. Two operations carry an explicit confirmation gate because the loss is not reversible from inside the conversation: **service deletion** and **wholesale service replacement** on a service with prior failed deploy history. Everything else the agent runs — deploys, env changes, lifecycle actions, restarts, scaling — runs without pausing for approval. Most of those are reversible by another call (a bad deploy is fixed by another deploy, a started service is stopped); a few are operationally destructive even though they aren't gated (deleting an env value, rotating a secret used by running services). You stay in the loop on these via the verify pass and the audit trail, not via a pre-call gate.

| Operation | Gate |
|---|---|
| **Service deletion** — removes the service entirely; the container, deployed code, env vars on the service, and (for managed services) the data inside it all go with it. | The MCP tool contract requires explicit user approval in the same conversation, by service name. Supported clients (Claude Code) enforce this. ZCP itself hard-blocks one case: deleting the hosted ZCP service it is running on (`SELF_SERVICE_BLOCKED`). |
| **Wholesale service replacement** (import override) on a service with prior failed deploy history | First call is refused with a structured payload naming the operation and target hostnames. The agent's second call must echo that payload back as a confirmation. |

Approval is an explicit yes paired with the service name in the same turn-window. Approval from a previous chat does not carry forward — a new conversation is a new approval window. The agent cannot delete on memory of an earlier yes, and ZCP cannot delete itself.

### Diagnose before destruct

When a service is in trouble — non-running state, failed last deploy, or both — ZCP enforces a single rule across destructive paths: read the platform's account of why before destroying anything. The first call against a service with failure history is refused; the agent reads the failure classification, events, and logs, then either fixes the cause (no destruction needed) or comes back with a confirmation that includes a summary of what it read.

The rule prevents two failure modes: **delete-and-retry loops** (the agent decides the cleanest fix is to delete and re-import, masking the cause every time) and **override-as-reset** (`zerops_import override=true` becomes a "make it like new" button that erases the failure context the next session needs).

Recovery state is not the same as broken state — a service that came up `READY_TO_DEPLOY` and is waiting for code is not in trouble. The gate fires on platform-recorded failure, not on idle.

For the broader threat model and what ZCP refuses outright (cross-project access, multi-project tokens, self-deletion), see [Trust model](/zcp/security/trust-model).

## Gotchas

1. **Multi-project tokens are refused at startup, not at first deploy.** A full-access token will not let ZCP boot at all. Generate a per-project token before connecting.
2. **`zcp init` overwrites `.mcp.json`.** If you added `ZCP_API_KEY` to the file and re-run `zcp init`, the regenerated template ships without your token. Re-add it before launching the agent.
3. **`GIT_TOKEN` is not `ZCP_API_KEY`.** `GIT_TOKEN` authorizes git push (hosted, when ZCP performs the push); `ZCP_API_KEY` authorizes ZCP against Zerops. Mixing them up either grants too much (a Zerops token used as a git push credential) or fails authentication. Treat them as separate surfaces.
4. **`ZEROPS_TOKEN` in GitHub Actions is a Zerops API token, not a GitHub PAT.** If a deploy from Actions fails on permissions and you reach for a GitHub PAT, you have rotated the wrong credential. The fix is in `gh secret set ZEROPS_TOKEN ...`.
5. **Rotation is picked up on next ZCP startup, not mid-session.** If you rotate the token while a Claude Code session is live, restart the agent so ZCP re-reads the env. The session in flight still holds the old value in process memory.
6. **Confirmation must be in the same conversation.** Approval from a previous chat does not carry forward to a new session. The agent cannot delete on memory of an earlier yes.
7. **A successful confirmation is still a destructive call.** Once you acknowledge replacement, the override runs. Roll-back is not automatic — recover lost data through [Backup](/features/backup), and review what changed through [Auditing agent work](/zcp/security/auditing-agent-work).
