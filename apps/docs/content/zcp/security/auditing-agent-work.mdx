Review Zerops Control Plane (ZCP) agent work through Zerops events, logs, deploy attempts, verification results, and git history. **Platform-side actions leave platform-native evidence; agent-side decisions, prompt content, and shell-driven file edits do not.** What you can audit and what you cannot are spelled out below — read the evidence-coverage matrix before you ask "did the agent do X" expecting the answer to live in Zerops.

This page assumes a development session has happened. Maybe the agent reported it finished. Maybe a deploy looped. Maybe you stepped away and came back to a long transcript. The platform recorded its share of the work either way — the audit is reading what's there, in the right order, and knowing what isn't.

## Evidence coverage — what is recorded, what is not

| Action | Where evidence exists | What's missing |
|---|---|---|
| Deploys (build status, appVersion, failure classification) | Zerops project events + deploy/verify history | Nothing — the platform record is canonical. |
| Runtime behavior (crashes, log lines, restarts) | Runtime logs scoped by service hostname, severity, time | Pre-deploy local stdout if your app prints before being deployed. |
| Lifecycle (start, stop, restart, scale, subdomain enable/disable) | Project events timeline | The agent's reasoning for choosing the action — that lives in the chat client. |
| Env-var changes (set, delete, update) | Project events; current values via discover | Per-call before/after diff is not separately recorded — read the discover snapshot before and after. |
| Service deletion and import override | Project events + the resulting service-stack changes | The structured `wouldDestroy`/confirmation exchange persists only in the chat-client transcript (same caveat as agent-side reasoning, below). Once executed, the destroyed state itself is gone — roll-back via [Backup](/features/backup), not the audit log. |
| Git-push delivery commits | Git history on the configured remote | Same as any other committer — `git log`, `git blame`, your PR review. |
| **Agent-side decisions, prompt history, planning** | The chat client's transcript | Not recorded by ZCP. If the chat is closed without export, that reasoning is gone. |
| **File edits the agent makes via a shell terminal in the workspace** | Filesystem state on the runtime container | Per-edit diff is not platform-recorded. Use `git status` in the runtime mount or commit incrementally to capture intent. |
| **Generated `.env` files (local agent bridge)** | The file in your project directory | ZCP wrote it from current project env; per-call detail of what was generated is not separately logged. |
| **Hosted code-server URL access** (who hit the workspace, when) | The Zerops platform's own access logs for the public subdomain | Not surfaced as part of the ZCP-side audit. See [Public access](/features/access). |
| **Browser-helper actions** in the hosted workspace (clicks, navigation) | The agent's chat transcript only | Not platform-recorded. |

Read the matrix before assuming an audit gap is suspicious — many gaps are platform boundaries, not silent failures. When the gap is load-bearing for your review (e.g., you need a tool-call audit for compliance), supplement the platform record with chat-client transcripts, git history, and your own commit discipline.

## Read in this order

The four platform-side streams have an order:

1. **Project events** — timeline of deploys, builds, scaling, status transitions. Scope by service hostname; the project-level view mixes everything.
2. **Runtime logs** — explain why a failure happened, scoped by hostname, severity, time, search.
3. **Deploys and verifies** — the canonical "did the agent ship something, and did it work" record. Deploy/build/appVersion events persist as durable platform records; verify results live in the tool response and ZCP's per-project work-session state, which is PID-scoped and cleaned over time. For durable verify audit, capture the response in the chat transcript or rerun verify on demand.
4. **Git history** (when delivery uses git-push) — commits the agent pushed, reviewed like any other diff.

Plus, when a session has gone sideways and you want the agent's current view of the project, **ZCP status** is what realigns: it rebuilds the picture from live platform state in one prompt.

Skipping events and starting from logs gives you a wall of stack traces with no framing.

## Review platform events

The project event timeline is the activity the platform itself recorded — every deploy, every build, every scaling action, every status transition. Ask the agent for it scoped to the service you care about:

```text
Show me the last 20 events for appdev.
```

```text
What happened on appdev between 14:00 and 15:00 today?
```

The agent reads the timeline back sorted by time, with each entry tagged by event type, action, status, and — for failures — a structured failure classification. See [Deployment lifecycle](/guides/deployment-lifecycle) for what each event type means in the build-and-deploy pipeline, and [Failure categories in Troubleshooting](/zcp/reference/troubleshooting#start-from-a-failure-category) for what the classification points at.

Filter by service hostname every time. The project-level view mixes events from every service and every old build container; for an audit, you want the runtime in scope and nothing else.

The timeline answers: did the agent deploy this service today (and how many times)? Did any of those deploys fail (and with what classification)? Did the agent restart, scale, or change the subdomain?

## Review runtime logs

Once events frame what happened, runtime logs explain why. Ask the agent for them scoped by hostname, severity, time window, and search text — you do not have to scroll a wall of output.

Useful prompts:

```text
Show me the ERROR-level logs for appdev in the last hour.
```

```text
Find any log line on appdev mentioning "migration" since 14:00.
```

```text
Tail appdev runtime logs since the last deploy.
```

Two distinctions to keep in mind:

- **Build logs vs runtime logs.** A failed build prints in the build container; a runtime crash prints in the runtime container. The agent picks the right stream for the failure classification it is looking at. If the timeline says `BUILD_FAILED`, ask for build logs; if it says `DEPLOY_FAILED` or the service crashed after a green build, ask for runtime logs.
- **Severity gates noise.** `ERROR` is the right starting filter for an audit. Pull `WARNING` next if you want a fuller picture; pull all severities only when you have a specific message in mind.

The logs are the application's own stdout and stderr. The agent does not paraphrase them in transit — what it shows you is what the runtime printed.

## Review deploys and verifies

Deploys and verifies are the canonical "did the agent ship something, and did it work" record. They persist on different surfaces: deploy/build/appVersion events are durable platform records you can reread weeks later; verify results live in the tool response and ZCP's per-project work-session state (PID-scoped, cleaned over time). For an audit you can revisit, capture the verify response in the chat transcript or rerun verify on demand.

Each deploy carries a build status, an appVersion status, and — for failures — a structured classification with a category, a likely cause, and a suggested next action. See [Failure categories in Troubleshooting](/zcp/reference/troubleshooting#start-from-a-failure-category).

Verification results sit alongside the deploys. Each verify run records service status, error log presence, and the HTTP probe outcome. A "successful" deploy with a failed verify is a real signal — the build finished and the runtime started, but the requested behavior did not pass. The audit needs both halves.

Useful prompts:

```text
Show me the last 5 deploy attempts for appdev with their outcomes.
```

```text
What was the failure classification on the appdev build at 14:32?
```

```text
Did the verify on appdev pass after the last deploy?
```

If a deploy looped (the agent retried the same build several times without progress), that pattern is visible in the timeline — repeated `BUILD_FAILED` entries with the same `failureCause`, no new commit between them. That is your cue to stop and read the cause yourself rather than ask the agent for another retry.

## Review git history (when close-mode is git-push)

When delivery uses `close-mode=git-push`, the agent's edits land as commits on a configured remote. At that point the audit shifts: the agent is just another committer, and the diffs review the same way a coworker's would.

Standard tooling applies. The agent is identified as the committer per your repository conventions; the commit messages and diffs are what they are. Treat them as you would any other change:

```bash
git log --oneline -20
git diff main...HEAD
gh pr view <number>
```

If the close-mode is `git-push` with a build integration (`webhook` or `actions`), each push fires the next deploy; the deploy and verify history above shows what the build did with that push. Cross-referencing the two streams is how you confirm "the commit landed and the deploy that ran from it succeeded". For the platform side of that integration, see [GitHub integration](/references/github-integration).

In `close-mode=auto` projects, no git evidence exists — the agent deployed directly through ZCP. In `close-mode=manual` projects, git review is your call: ZCP recorded the deploys it ran at your request and yielded delivery to you, so any commits are yours, not the agent's.

## Take over from the agent

Auditing usually leads to a decision: keep the agent running, take the rest manually, or roll back. The take-over path differs depending on whether the session is still in flight or already closed.

**Mid-session — agent looks stuck.** Before changing anything, get the agent's current view from ZCP:

```text
Read ZCP status before doing anything else.
```

ZCP status is the canonical "what does the agent see right now" call. It returns the current scope, the last actions, and what is still pending — rebuilt from live platform state plus the agent's session record. Compared to scrolling the transcript, this reads from reality. If the agent's chat-side picture has drifted (a long session, a closed tab, a restarted machine), status is what realigns it. See [How ZCP works](/zcp/concept/how-it-works) for the recovery model.

From there:

| Situation | Action |
|---|---|
| Agent has a clear next step but is hesitating | Tell it to proceed. Status told you the path is intact. |
| Agent is looping on the same failure | Stop the loop. Read the failure classification and last logs yourself, give the agent the corrected approach, or take over manually. Five attempts without progress is the standard stop signal — see [Build and verify an app](/zcp/workflows/build-and-verify-app). |
| Scope is wrong (agent targeting `appstage` when you asked for `appdev`) | Correct the scope explicitly. The agent picks up the new value on the next call. |
| Session has drifted past usefulness | End the session. The deploys and verifies it ran are recorded; you can resume from the same project on a fresh start. |

**After a normal close — keep, take over, or revert.** A session that auto-closed already has a clean deploy and a passing verify on every service in scope. The audit decides whether to keep, take over, or roll back:

| Decision | What to do |
|---|---|
| Keep the changes | Nothing further is required. The deploys are live, the verify passed. |
| Take over manually for the next change | Switch the close-mode to `manual` if you want ZCP to stop initiating deploys, or just keep going yourself. See [Choose how finished work ships](/zcp/workflows/delivery-handoff). |
| Revert via git | When `close-mode=git-push`, revert the commit on the remote and let the build integration ship the previous state. Standard `git revert` workflow. |
| Restore from backup | When data was lost (a destructive operation, a bad migration), use the platform's [backup and restore](/features/backup) — that is the data-safety net, not ZCP. |

Before any irreversible follow-up — a destructive import override, a service deletion — re-read [Destructive actions and confirmations](/zcp/security/destructive-actions). The audit you just ran is exactly the diagnosis those gates expect you to have done first.

## Gotchas

1. **Filter events by service hostname every time.** The project-level view mixes builds from every service and every retry; an audit needs the runtime in scope and nothing else. Without the hostname filter you will read someone else's failure and diagnose the wrong thing.
2. **Read status before changing anything when taking over.** The chat transcript is not the source of truth. Status rebuilds the agent's view from live platform state — a single call confirms whether the scope, last action, and pending work match what you remember. Acting on the transcript without checking status is how take-overs break things the agent had right.
3. **Build logs and runtime logs are different streams.** A failed build never appears in runtime logs; a runtime crash never appears in build logs. The timeline's failure category points at the right stream — `build` → build logs, everything else → runtime logs. Pulling the wrong stream wastes the audit.
4. **The agent's diffs are like any committer's.** When `close-mode=git-push`, do not treat the agent's commits as a separate review surface. Use `git log`, `git diff`, your normal PR review. The fact that an agent authored the change does not make the diff special.
5. **Backup is the data safety net, not ZCP.** Audit and roll-back via git cover code; if data was lost, [Backup](/features/backup) is what gets it back. Do not expect the audit trail to undo a destructive database operation — it can show you that one happened, not undo it.
