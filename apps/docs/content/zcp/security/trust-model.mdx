Zerops Control Plane (ZCP) gives a coding agent project-scoped power inside one Zerops project. Same control plane, two ways to run it — and the blast radius is different. That difference is the headline of this page.

:::caution Public preview
Zerops Control Plane (ZCP) is in public preview. Use it for development and staging projects, keep production in a separate Zerops project, and expect setup details to change.
:::

## Container or local: two different blast radii

| | Hosted workspace | Local agent bridge |
|---|---|---|
| Where the agent runs | A `zcp` service container in your Zerops project | Your laptop |
| What the agent can touch | The container workspace, project services, project token, and (when requested) runtime filesystems via SSHFS mounts. | The same project surface, **plus whatever the agent client can reach on your laptop**. |
| Network reach | The project's private network. | Project services over VPN; everything else your laptop can already reach. |
| Safety profile | **Safe by design.** Isolation is structural — the container has no path to your laptop, your home directory, or your other Zerops projects. | **Supervise the agent client.** ZCP itself is project-scoped, but the agent process inherits your user. What the client does with file edits, shell commands, and other tools follows your client settings, not ZCP. |

This is not an argument against the local path. The local bridge is the right choice when you want to keep your editor, dotfiles, and toolchain. It is a different security model — recognize it, set the agent client's permissions accordingly.

For per-action gates (delete, override import), see [Destructive actions and confirmations](/zcp/security/destructive-actions). For token storage and rotation, see [Tokens and project access](/zcp/security/tokens-and-project-access).

## Project is the boundary

One ZCP process works against one Zerops project. The boundary starts at authentication: at startup, ZCP resolves the token to a project before it exposes operations to the agent. A token with access to no projects is refused; a token with access to multiple projects is refused. The intended shape is a Zerops token scoped to exactly one project.

ZCP does not create a separate permissions system. Zerops [RBAC](/features/rbac) remains the authority. If the token can deploy, manage env vars, read logs, or operate services in that project, ZCP can expose those operations to the agent. If Zerops rejects the token for an operation, ZCP cannot bypass that rejection.

| Question | Boundary |
|---|---|
| What project can ZCP see? | The one project resolved from the token at startup. |
| What resources can the agent change? | Anything the token can change inside that project: runtime services, managed services, env vars, deploys, logs, lifecycle, scaling, public access. |
| What is outside reach? | Other Zerops projects, organization-wide settings, any resource Zerops RBAC does not grant the token. |
| Where is the network scoped? | The project's private network. See [Public access and private networking](/features/access). |

Project-scoped does not mean read-only. A full project token is still powerful inside that project — use it like a project-level operations credential: give it to dev/staging projects where agent work is intended, rotate it when needed, keep it out of repositories.

For token storage and rotation, continue to [Tokens and project access](/zcp/security/tokens-and-project-access).

## Hosted workspace specifics

The hosted workspace is the safe-by-design path from the table at the top. A few specifics worth naming:

- **`zcp` service ≠ runtime service.** The ZCP service is the control surface; runtime services are where your app code runs. Deploys target a runtime service, not the `zcp` service.
- **Filesystem reach is narrower than network reach.** The agent reaches every project service over the private network, but only sees runtime files through SSHFS mounts when requested. `/var/www` is the SSHFS mount root, not an app repository.
- **A terminal in the hosted workspace has the same project-scoped power as the agent.** That is convenient, and it is the reason the preview guidance keeps production in a separate project.

## Local agent bridge specifics

The local bridge is the supervise-the-client path from the table at the top. A few specifics:

- **Per-project files are isolated.** Each project directory has its own `.mcp.json` and `.zcp/state/`. Switching projects means switching directories — ZCP does not merge local state between projects.
- **VPN setup is deliberately outside ZCP's authority.** Bringing up the Zerops VPN requires admin or root approval on your laptop. ZCP can tell you the `zcli vpn up` command but cannot start it for you.
- **`.env` is generated, not synced.** ZCP writes a snapshot to your project directory when you ask for it. It is not a live link — regenerate after env changes.
- **The local bridge does not protect your checkout from the agent client.** ZCP's boundary is the Zerops side; your client's permissions decide what happens on your laptop.

For full local setup, see [Use ZCP from your laptop](/zcp/setup/local-agent-bridge).

## What ZCP refuses by design

ZCP uses hard refusals for boundaries that should never be inferred, and confirmation gates for actions that are destructive but sometimes necessary. The important part is that these behaviors are concrete, not aspirational.

| Behavior | When it triggers | Result |
|---|---|---|
| Refuse a token with no project access | Startup cannot resolve any Zerops project from the token. | ZCP does not start for agent operations. Use a token with access to one project. |
| Refuse a multi-project token | Startup sees more than one accessible Zerops project. | ZCP does not choose for the agent. Generate a project-scoped token or scope `zcli` to one project. |
| Refuse hosted self-deletion | `zerops_delete` targets the service where ZCP is currently running. | The call is blocked. Delete that service manually through Zerops UI or `zcli` if you really intend to remove the control plane. |
| Require named approval for deletion | An agent wants to delete a service. | The destructive tool contract requires explicit approval in the current conversation for that specific service name. The agent should not delete proactively. |
| Gate destructive service replacement | `zerops_import override=true` would replace services with prior failed build or deploy history. | The first call returns a structured `wouldDestroy` payload. A second call must include matching `confirmDestructive` fields before the import proceeds. |

The import gate is intentionally not a permanent blocker. Replacing a failed service can be the right recovery path. The gate makes the loss explicit first: which service stacks are being replaced, which targets are acknowledged, and which operation is being confirmed.

For the detailed rules, see [Destructive actions and confirmations](/zcp/security/destructive-actions).
