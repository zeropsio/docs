Choose whether Zerops Control Plane (ZCP) closes work by deploying directly, pushing to git, or handing off to your CI. This page picks up after [Build and verify an app](/zcp/workflows/build-and-verify-app) — the first deploy worked, verification passed, and the app does what you asked. Now decide how the next change gets out the door.

If you don't have a CI pipeline yet, the third answer below is not for you — pick "the agent keeps deploying" and revisit when your team needs git-tracked releases.

ZCP projects are intentionally dev/staging environments. Today, packaging a verified ZCP project for production handoff is supported via the [export workflow](/zcp/workflows/package-running-service); production then lives in its own Zerops project deployed by your CI or a release pipeline.

## Three answers

- **The agent keeps deploying directly.** Future changes ship the same way the first deploy did — straight from ZCP through the Zerops [build and deploy pipeline](/guides/deployment-lifecycle). Right for solo work and fast iteration.
- **Push to git, then build from there.** Future changes get committed and pushed to a configured remote. Either Zerops or your existing CI takes the push and runs the build. Right when the team's source of truth is git and you want reviewable commits.
- **Hand off to your CI or a human.** ZCP records every deploy and verify the agent runs at your request, but does not initiate further deploys. Right when an external pipeline or release process owns delivery from here.

These are recorded against each service. The agent translates "set the close-mode" or "configure git push" prompts into the right operations; you do not call them by name.

The first deploy always uses the direct path regardless of the answer you pick later — there is no deploy yet for the choice to apply to.

```text
Set up git-push delivery for appdev. Push to git@github.com:my-org/notes-app.git.
```

```text
Keep ZCP deploying appdev directly for now.
```

```text
ZCP records changes for appdev, but my CI takes over from here.
```

## The agent keeps deploying

The deploys ZCP already ran during iteration are how the next change ships, too. No further configuration. A verified session can close on the deploys that already landed.

Pick this when the project is small or solo, when the deploys ZCP runs are the canonical way the app updates, and when you do not need a git-tracked history of the deploys themselves.

## Push to git, then build from there

Future changes get committed and pushed to a configured git remote. Whatever sits behind that remote — a Zerops-managed build or your own CI — produces the next deploy.

This requires committed code. ZCP does not auto-init git or invent a commit on your behalf. Before git-push delivery becomes useful you need an initialized repository with at least one commit covering the work you want to ship, a remote URL ZCP can push to, and credentials that authorize the push.

When you ask the agent to set up git-push delivery, it provisions the remote URL and credentials so future pushes work without per-call configuration. If the setup step fails — repository unreachable, credential missing, working directory has no commits — the agent reports the specific reason. Fix the underlying issue and ask again.

How the credential is sourced depends on the workspace:

- **Hosted workspace.** If the project already has `GIT_TOKEN` in its env, the agent reuses it. Otherwise, provide a token that can push to the remote — typically a fine-grained GitHub or GitLab access token scoped to the target repo.
- **Local agent bridge.** The push runs from your laptop, so your local git credential helper handles authentication — SSH keys, the macOS keychain, `gh auth login`, whatever you already use. ZCP does not need a token of its own. See [Tokens and project access](/zcp/security/tokens-and-project-access).

### What fires after a push lands

- **Nothing tracked by ZCP.** Whatever your repository already has — a non-Zerops CI, a deploy hook elsewhere — keeps running. ZCP records the deploys you ask it to verify and stays out of the way.
- **The Zerops dashboard build integration.** The Zerops dashboard pulls the repository and runs the [build and deploy pipeline](/guides/deployment-lifecycle) — see [GitHub integration via the dashboard](/references/github-integration#integration-via-zerops-gui).
- **A GitHub Actions workflow you check in.** Your repository's `.github/workflows/zerops.yml` checks out the code and pushes it to Zerops via `zcli` — see [GitHub Actions](/references/github-integration#github-workflow-integration).

The GitHub Actions option needs a **Zerops API token** stored as a GitHub Actions secret (it authenticates `zcli` against Zerops; it is **not** a GitHub token). One credential, one rotation surface — you can reuse the project-scoped token already behind `ZCP_API_KEY`. The agent prepares the right `gh secret set ZEROPS_TOKEN ...` command for your environment. Token-shape details: [Tokens and project access](/zcp/security/tokens-and-project-access).

## Hand off to your CI or a human

ZCP keeps recording every deploy and verify the agent runs at your request, but does not initiate further deploys on its own and does not auto-close the session.

Pick this when an external CI/CD pipeline owns delivery, a human takes over for a release step that does not belong inside the agent loop, or the next change is ambiguous enough that explicit close calls beat any automation.

In this mode, ending the session is a deliberate step. The agent does not auto-close it for you, even after a clean deploy and verify.

## Change later

None of these choices are sticky. The handoff style, the git-push capability, and any build integration are read fresh on every handoff. Switch any one of them at any time:

- Move from direct deploy to git-push once the project is large enough to want commits.
- Add a Zerops dashboard or GitHub Actions integration after git-push is already in use.
- Drop back to manual hand-off for a release where you want full control.

## Gotchas

1. **First deploy is always direct, regardless of the choice.** Setting up git-push delivery early does not redirect the first deploy to git. The choice governs subsequent changes, not the deploy that closed the previous workflow.
2. **GitHub Actions secret is a Zerops API token, not a GitHub token.** The workflow runs `zcli` against Zerops; it needs a Zerops API token in `ZEROPS_TOKEN`. If a deploy from Actions fails on permissions and you reflexively rotate your GitHub PAT, you have rotated the wrong credential.
3. **Build integration set to "none tracked" does not mean no build will fire.** It means ZCP did not set one up. Whatever your repository already has — a non-Zerops CI, a deploy hook elsewhere — keeps running.
