description: "Use ZCP to create new Zerops services or adopt existing ones before any application work starts."

Use Zerops Control Plane (ZCP) to create new Zerops services or adopt existing ones before any application work starts. This is the first workflow because the agent needs a real runtime target before it can build or deploy anything.

You do not pick a route. Ask for the project shape you want; the agent reads the project state and explains the choice it made.

## What this workflow does

Create-or-adopt is **infrastructure only**. It answers:

- Which runtime services will hold application code?
- Which managed services does the app depend on?
- Are the services already present, or should ZCP create them?
- Are the services running and visible to ZCP?

The boundary is strict. This workflow does not write application code, create `zerops.yaml`, run a deploy, or verify behavior. When any of those are needed, the work moves to [Build and verify an app](/zcp/workflows/build-and-verify-app).

The `zcp` service is the workspace and the agent connection — not the application. Runtime services are where your code runs (`appdev`, `appstage`, `api`, `web`, and so on). If the agent starts treating `zcp` as the app runtime, stop and correct the target.

## Two starting situations

| Situation | What the agent should do |
|---|---|
| **The project already has runtime services.** This includes any project created from a Zerops recipe. | Adopt the existing runtimes. Identify which are runtime services and which are managed dependencies. Do not recreate or rename services just to begin. |
| **The project is empty from the application point of view** (only the `zcp` service, or nothing yet). | Create the runtime and managed services. The agent may use a Zerops-published starter (recipe) when the request matches one, or a custom plan when it does not. Either way the agent explains the choice and creates the services before any code work. |

If a setup was interrupted earlier and you returned, ask the agent for current ZCP status before changing anything else. ZCP rebuilds the picture from live platform state and resumes from the last known step.

## Prompts that work

Write the prompt in terms of the project shape you want. Name a runtime hostname only when you already know it.

```text
Create the services for a Node.js API with PostgreSQL. Use appdev and appstage.
```

```text
Use the existing Laravel services and build a small notes app.
```

```text
Build me a small team dashboard. Pick a suitable starter and explain what you chose.
```

The agent answers with: which path it took (adopt vs create), the target services, and the next confirmation it needs.

## Stop conditions — when create-or-adopt is done

The agent should report this workflow complete when:

- Runtime services are identified and separated from managed dependencies.
- New services, if any, were created in Zerops; existing services, if any, were adopted instead of recreated.
- Managed services have exposed env-var keys for the runtimes to reference later.
- No application code was written.
- No `zerops.yaml` was authored for the app.
- No deploy ran.

When the next task involves files, framework setup, `zerops.yaml`, deploys, logs, or behavior checks, continue to [Build and verify an app](/zcp/workflows/build-and-verify-app).

## Common failures and what they mean

| Symptom | What it usually means | What to do |
|---|---|---|
| The agent wants to deploy `zcp`. | It confused the workspace service with the app runtime. | Stop and name the real runtime service. |
| The agent treats existing services as if they need to be created. | It is taking the create path on a project that should adopt. | Ask it to inspect the project first and adopt what is already there. |
| The agent writes application files during this workflow. | It crossed into application work too early. | Stop and finish service setup, or move to [Build and verify an app](/zcp/workflows/build-and-verify-app). |
| The agent creates a new service with a hostname that already exists. | The plan did not account for current project state. | Ask it to rediscover services and revise the plan. |
| A service-creation step fails. | The infrastructure request did not complete cleanly. | Treat as a hard stop. The agent should explain the failure and ask for a decision; it should not retry automatically. |
| The agent asks you to describe every service manually. | It has not used live project discovery yet. | Ask it to inspect the project first and explain what it found. |

Create-or-adopt failures do not have retry magic. Provisioning infrastructure twice without understanding the first failure can create hostname conflicts or partial service state.
