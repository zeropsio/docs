In this guide, we'll go from zero to a live, production-ready app with a managed database, a public URL, and auto-deploy on every git push. Takes about 10 minutes.

You'll deploy a working recipe first, then swap in your own code in Step 7.

By the end of this tutorial, we'll have:

- **A real app running live** with a public URL
- **A managed database** with no setup, no connection strings, no SDK
- **Auto-deploy on every git push** so pushing to main builds and deploys automatically
- **All services on a private network** talking to each other by hostname, zero config

:::tip Before we start
Sign up at [app.zerops.io](https://app.zerops.io) and you get **$65 in free credits**, no credit card needed. A simple app with a database costs roughly $3-5/month, so credits go a long way.
:::

## Pick your path

This guide follows the recipe path, the fastest and most reliable way to get started on Zerops. Recipes are working apps with the right infrastructure already configured. We'll deploy one, then swap in our own code.

<Tabs>
<TabItem value="stack" label="🎯 I know my stack" default>

Pick a recipe matching your tech stack. Next.js, Node.js, Python, Go, Laravel and more are all available.

On the welcome screen, find your stack in the **recipe grid on the left** and click it.

</TabItem>
<TabItem value="showcase" label="👀 Show me what Zerops can do">

Deploy the showcase recipe, a full multi-service app with a frontend, background worker, PostgreSQL, Valkey, NATS and S3.

On the welcome screen, click **Deploy showcase-recipe-small-prod** at the top.

</TabItem>
<TabItem value="existing" label="🤖 I have an existing app">

Create a project with ZCP. Zerops' AI agent will look at your codebase and bootstrap the right config for you.

See the [ZCP guide](/zcp/quickstart) for a dedicated walkthrough. The rest of this guide covers the first two paths.

</TabItem>
</Tabs>

:::note Want to start from a blank project?
You can, but we'd recommend starting with a recipe even if you plan to replace all the code. Recipes give you a correct `zerops.yaml` and infrastructure config as a starting point. Doing it from scratch means figuring all of that out yourself, which takes a while.
:::

## 1. Sign up and land on the welcome screen

Head to [app.zerops.io](https://app.zerops.io) and create an account.

After signing up, you'll land on the welcome screen with three sections:

- **Top center**: the Zerops showcase recipe, a full multi-service app you can deploy in one click
- **Left side**: a recipe grid with all supported stacks
- **Right side**: a manual project creator for when you want full control or want to add ZCP

For the first two paths, find your stack in the recipe grid on the left and click it. For the showcase, click **Deploy showcase-recipe-small-prod** at the top.

---

## 2. Choose your recipe variant and deploy

After clicking a recipe, you'll see the available variants for that stack.

For example, clicking **Next.js** shows two options:

- **Next Static Hello World**: a static export served by Nginx, no database. Good for frontend-only apps.
- **Next SSR Hello World**: server-side rendering with a PostgreSQL database. Good for full-stack apps.

Pick the one that matches what you're building. You'll see a full breakdown of what gets created, services, resource sizes, and the exact price before committing anything.

Select **Small Production** and hit **Deploy**.

:::info What is a recipe exactly?
A recipe is a working app with infrastructure already configured: managed database, environment variables, `zerops.yaml`, everything connected and ready. It's a correct starting point, not a finished product. We'll swap the code for our own later.
:::

---

## 3. Watch the pipeline run

After hitting Deploy, you'll land on the main dashboard. On the right side you'll immediately see a pipeline running, no need to navigate anywhere.

You'll see it go through these steps in real time:

1. Initializing build container
2. Run build commands from zerops.yml
3. Create app version and upgrade service
4. Done

:::tip Good to know
The build container is **temporary and free**. Zerops spins it up, builds the app, saves the output, and deletes it. You're only charged for the running app.
:::

Once it's done, your project is live.

At this point, your app is running privately; next we make it public.

---

## 4. Look at what just got created

Click into your new project. Depending on the recipe you picked, you'll see something like:

- **app**: your app service (static or SSR, depending on what you chose)
- **db**: a managed PostgreSQL database, already running (SSR recipes only)
- **Project core**: load balancer, firewall, logger, all managed by Zerops, always there

All of these are already talking to each other on a private network. Nothing to configure.

:::tip The private network moment
Coming from Vercel and Supabase, you're probably used to copying long database URLs and pasting them as environment variables. On Zerops, your app connects to the database with just `DB_HOST: db`. That's it. No IP address. No connection string. It's on the same private network. This is the thing that catches most people off guard the first time.
:::

---

## 5. Make it public

The app is running but not publicly accessible yet.

1. Click into your app service
2. Click **Subdomain, domains and IP access**
3. Toggle on the Zerops subdomain

You'll get a live URL that looks something like `app-2437-3000.prg1.zerops.app`. Open it and your app is on the internet.

When you're ready for production, you can point a custom domain at it. Zerops handles the SSL certificate automatically.

---

## 6. (Optional now) Connect GitHub and auto-deploy

Now let's connect a GitHub repo so every push deploys automatically.

1. Click into your app service
2. Go to **Pipelines and CI/CD settings**
3. Click **Connect with a GitHub repository**
4. Authorize Zerops and select your repo and branch (usually `main`)
5. Check **"Trigger once after the activation?"** so the first deploy kicks off right away
6. Click **Activate pipeline trigger**

From now on, every push to main triggers a build and deploy automatically. Zero downtime, Zerops runs the new version alongside the old one, waits for a health check, then switches traffic over.

---

## 7. Make it yours

Now let's replace the recipe code with our own.

<Tabs>
<TabItem value="replace" label="Option A: Replace the code" default>

Fork the recipe's GitHub repo, swap the app code with your own, and push. The `zerops.yaml` and all the infrastructure stay exactly the same.

1. Fork the recipe repo from [github.com/zeropsio](https://github.com/zeropsio)
2. Replace the app code
3. Connect your fork to Zerops (same steps as Step 6)
4. Push and it deploys automatically

</TabItem>
<TabItem value="integrate" label="Option B: Apply patterns to your existing codebase">

Read the recipe README. There's always a section called something like "Integrating your own app" that tells you exactly what to add.

Usually it's just:

- Add a `zerops.yaml` to the repo root, copying from the recipe
- Adjust the build and run commands for your stack
- Connect your repo (Step 6) and push

:::tip Next.js developers, one thing to know
Add `output: 'standalone'` to your `next.config.mjs` before deploying. This bundles everything Next.js needs into `.next/standalone` so you don't need to ship `node_modules` separately. The Next.js SSR recipe already does this, so you can copy the pattern directly from it.
:::

</TabItem>
</Tabs>

If you already have a repo, use Option B; if you're starting fresh, use Option A.

## If something breaks

Got a 502 or an app crash on startup? Start here.

### Check the runtime logs first

Dashboard, click your app service, click the three-dot menu, then **Runtime log**. The error will be right there, usually in the last few lines.

Two things come up most often on a first deploy:

<Tabs>
<TabItem value="module" label='"Cannot find module"' default>

Your `node_modules` aren't being deployed. Either:

- Add `node_modules` to `deployFiles` in `zerops.yaml` (quick fix)
- Or use `output: standalone` in `next.config.mjs` (better for Next.js, bundles everything you need)

</TabItem>
<TabItem value="db" label='"Database connection refused"'>

Your environment variables aren't set correctly in `zerops.yaml`. Copy this exactly:

```yaml
DB_HOST: db
DB_PORT: "5432"
DB_USER: ${db_user}
DB_PASSWORD: ${db_password}
DB_NAME: db
```

The `${db_user}` and `${db_password}` syntax is how Zerops pulls in the credentials it auto-generates for your managed database.

</TabItem>
</Tabs>

:::tip Debug locally with VPN
Install zcli first (see [CLI reference](/references/cli)), then run `zcli vpn up [your-project-id]` and your laptop joins the project's private network. You can then connect to `db:5432` directly from your local machine using TablePlus, psql, or any database client. Disable SSL when connecting over VPN, security is handled by the tunnel itself. If `db` doesn't resolve, try `db.zerops` instead.
:::

You now have a live app, managed database, and auto-deploy. Next steps:

- **[Custom domain](/references/networking/public-access)**: add your domain, SSL is automatic
