---

Coding agents need somewhere to operate, not just somewhere to generate code. The hard part is rarely "write the function" — it's where the agent builds, deploys, tests, and iterates against real infrastructure across hours and sessions, with state that persists and a path to production.

ZCP is built for that. It's a real Linux dev container deployed inside your project, on the same private VXLAN as your services, with SSHFS access to your code, the Zerops MCP server, and `zcli` so anything inside can deploy, scale, or inspect the project. Optionally, an agentic CLI like Claude Code is bundled in, and a browser-based VS Code (the Cloud IDE) gives you a graphical surface alongside SSH and the web terminal.

:::caution Designed for development, not production
ZCP carries a project-scoped integration token with full rights and runs with relaxed isolation defaults so the agent can read and modify the services you select. It belongs in a development project, not a production one.
:::

This page assumes you've read [Local & Remote Development](/features/local-remote-development). ZCP uses the same access methods (web terminal, Cloud IDE, SSH, VPN), the same hostnames, and the same deploy pipeline.

In a typical setup, ZCP lives in a development project alongside the dev (and optionally stage) runtimes the agent operates on. Production is a separate project — the agent's work reaches it through the delivery contract chosen at the end of a session, not through direct deploy from ZCP.

That separation is a security boundary, not bookkeeping. The integration token ZCP carries is scoped to its own project; it cannot reach across to another. The agent-deletes-prod-database stories that have become a fixture of agentic-tooling discourse hit the same wall on Zerops: the project boundary. The worst case of an agent going off the rails in your dev project is dev data, not production traffic.

## What ZCP is

ZCP is a service of type `zcp@1` — Ubuntu, two GB minimum RAM, a single container per service. You add it to a project the same way you add any other service: from the project dashboard or a recipe. When you do, three things happen on deploy:

- An integration token scoped to the project with full rights is created and injected into the container.
- The container is deployed with `envIsolation: none` and `sshIsolation: vpn service@zcp`, so the workspace can read and modify the services you select.
- The base image is provisioned with a curated dev toolchain.

That toolchain is what you'd want on a working dev box: `git`, `gh` (GitHub CLI), `jq`, `yq`, `ripgrep`, `fd`, `fzf`, `bat`, `tree`, `tmux`, `htop`, `ncdu`, `httpie`, `make`, `psql`, `mysql`, `redis-cli`, `chrome`, `puppeteer`, an agent-controllable browser, `sshfs`, `zsh` with Oh My Zsh, and `zcli`. You can install whatever else you need with `apt`, `pip`, or `npm` — it's a real Linux box, not a sandbox.

ZCP being a normal service has consequences worth noticing. You can SSH into it. You can install packages. You can run multiple processes. You can mount additional services. You can fork the base image and bake your own tools in. The MCP server and the agent are two of the things that come preconfigured — not the platform itself.

## What you configure

Two toggles on top of the base ZCP, both optional:

**Coding Agent.** Bundles an agentic CLI inside the container, configured with auth and MCP wiring. *Claude Code is the only agent currently bundled* — auth, MCP, and orchestration flows are optimized for it; support for additional agents is in the works. You authenticate the agent in one of two ways:

- **Subscription Login.** OAuth into your Anthropic subscription. The container prompts you to sign in once when the service becomes active.
- **API Token.** A personal API key stored in the container's secret env vars.

**Cloud IDE.** Browser-based VS Code with the Claude Code plugin preinstalled. The plugin gives you an interactive surface alongside the agent's terminal CLI — useful when you want to watch the agent work and step in occasionally rather than running it autonomously over SSH. Access is configurable:

- **VPN Only.** Reachable through `zcli vpn up`, no public exposure. Most secure.
- **Public Access.** Reachable on a `.zerops.app` subdomain, gated by an auto-generated password.

ZCP without either toggle is just a Linux dev container on your project's network with `zcli` and the MCP server, accessible via SSH and the web terminal — useful as a remote workstation for humans (see [Local & Remote Development](/features/local-remote-development)).

## Starting from a recipe

Configuring ZCP from the toggles is one path. The other is starting from a [Zerops recipe](https://app.zerops.io/recipes) that ships with an AI Agent environment preconfigured.

The catalogue isn't limited to `hello world` setups. It includes production-shaped applications — multi-runtime stacks with managed databases, cache, search, queues, and object storage already wired together on a private network. Pick the AI Agent variant when you create the project and the recipe deploys with ZCP, the Cloud IDE, and the dev runtimes for that stack already in place. The agent can start on actual application logic in minutes, against real managed services on the project's network.

The catalogue keeps growing — new stacks ship regularly, and existing recipes get the AI Agent variant added as the agent flows mature for that runtime. Recipes are a starting point, not a constraint. The resulting project is yours — same `zerops.yml`, same services, same private network. Edit, extend, or replace any of it.

## Why transparent infrastructure works for agents

Agents reason over text. The choices Zerops makes for human transparency are also what makes the platform legible to agents:

- `zerops.yml` is structured and documented. The agent reads, modifies, and commits it the way it would any other config file.
- Resource allocation is queryable. The agent can compute what a change costs before scaling.
- Service topology is explicit — services have names, ports, and dependencies the agent can reason about directly.
- Logs, events, and deploy history are streamed. The agent can verify outcomes, not guess at them.
- Dev, staging, and production share infrastructure. The agent's deploy to staging exercises the same pipeline production uses.

The MCP server is the ergonomic layer on top. It tightens the agent's loop and gives host CLIs a permission surface to gate against — but it doesn't expose anything that isn't also reachable through `zcli` and SSH. If MCP didn't exist, an agent on ZCP would still be more capable than one in a sandbox. It would just be slower to find the right tool.

## The agent's working loop

A session inside ZCP runs in three phases. Each is a contract — the agent commits to it, and you can audit against it.

### Bootstrap

Before any code change, the agent establishes what's already in the project. One of four routes:

- **Adopt.** App runtime services exist. The agent maps them, mounts the relevant ones, and works against them. The most common case, including projects already created from Zerops recipes.
- **Recipe.** The project is empty (or contains only ZCP itself), and the user's request matches a known stack. The agent starts from a Zerops recipe and adapts it.
- **Classic.** The project is empty and the request needs a custom service plan. The agent designs the plan, creates the services, writes the initial `zerops.yml`.
- **Resume.** A previous session was interrupted mid-bootstrap. The agent reconstructs state rather than starting over.

The agent picks one and tells you which.

### Develop

The actual change. The agent modifies code in the in-scope dev runtime via SSHFS, the dev server hot-reloads, and the agent tests against real services on the private network — the same managed Postgres your production will use, the same Redis, the same S3-compatible storage, just at smaller resource levels. Because dev and production share infrastructure, code that works against dev's `db:5432` is already proven against the kind of database production runs.

Failures are categorized — `build`, `start`, `verify`, `network`, `config`, `credential` — and retries use new evidence rather than the same attempt with hope.

Verification has two layers. The runtime is reachable (a platform-level health check). The requested behavior works (an application-level check the agent runs against the actual endpoint, UI, or stage URL). A green deploy with a 500 on the relevant route is not done.

### Delivery

After a verified deploy, the agent picks a delivery contract for future changes:

- `auto` — finished work stays on the direct Zerops deploy path.
- `git-push` — work is committed and pushed to a configured remote, with the resulting build observed via webhook or GitHub Actions.
- `manual` — a human or external CI takes over.

Delivery is decided after a working deploy, not before. It's a contract for what comes next, not a flag that affects the current work.

## The MCP tool surface

MCP tools fall into three categories, tagged so host agents can gate permissions:

| Tag | Example tools | Behavior |
|---|---|---|
| Read-only | `zerops_logs`, `zerops_events`, `zerops_discover`, `zerops_verify`, `zerops_knowledge` | Inspect state and configuration. Safe to call without confirmation. |
| Destructive | `zerops_deploy`, `zerops_scale`, `zerops_env`, `zerops_delete`, `zerops_manage` | Mutate the project or its services. Host agent should require confirmation. |
| Operational | `zerops_recipe`, `zerops_mount`, `zerops_subdomain`, `zerops_workflow`, `zerops_dev_server` | Set up infrastructure or coordinate work. Host agent decides based on context and policy. |

Host CLIs read these tags and apply your configured policy — typically "auto-allow read-only, require approval for destructive, ask once per operational tool." Tags also matter for non-interactive use: integration tokens can be scoped to read-only operations for CI jobs, while interactive sessions get the full surface with approval prompts gating mutations.

## Source control inside ZCP

The agent runs in a container, not on your laptop, so its git access has to be set up on the container side. Two patterns work:

- **GitHub via `gh`.** The GitHub CLI is preinstalled. Run `gh auth login` once (interactive subscription login or token paste). `gh` handles HTTPS auth, PR creation, and review flows; the agent can use it directly.
- **Token in env vars.** Save a fine-grained personal access token to ZCP's secret environment variables and `git` reads it through a credential helper. Useful for non-interactive setup or scripted bootstraps.

For developers who'd rather use SSH agent forwarding from their laptop to ZCP, that pattern works too — see [Local & Remote Development](/features/local-remote-development#native-ide-over-ssh).

## Permissions, isolation, and audit

Two ZCP-specific isolation settings make the agent's work possible:

- **`envIsolation: none`** — ZCP can read environment variables from other services in the project. This is what lets the agent connect to your databases without you copying credentials around.
- **`sshIsolation: vpn service@zcp`** — ZCP can SSH into the services you select for it. This is what makes SSHFS-mounted dev work.

Both are scoped to ZCP itself; other services in the project keep their normal isolation defaults. The integration token injected into the container is what lets MCP and `zcli` operate on the project. Treat it like any other privileged credential: ZCP is the right tool for trusted developer or agent use, not for hosting public traffic or untrusted code.

For audit, you have the same surface a human session leaves behind: deploy history, events stream, log search, and the project's git history if `git-push` delivery is in use. Project roles (Owner, Admin, Developer, Guest) apply to ZCP itself — see [Roles & Permissions](/features/rbac).

## Hacking on ZCP

ZCP is a normal Zerops service running the `zcp@1` Ubuntu base image. The configuration generated when you toggle Coding Agent and Cloud IDE is a starting point — not a sealed product. You can read the generated `zerops.yml`, edit it, and redeploy.

A few things this means in practice:

- SSH into ZCP and `apt install` whatever else your team needs.
- Edit the generated `zerops.yml` to add `initCommands`, custom env vars, or additional `startCommands`.
- Fork the base image and ship a hardened or extended version with your team's standard setup baked in.
- Run additional processes alongside the agent and Cloud IDE — a pre-warmed cache, a custom inspector, your own MCP server.

If your team has a "developer onboarding script" worth a few hundred lines, it can live in ZCP's `initCommands` and apply to every dev workspace and every agent session automatically.

## How this differs from related tools

The agent space contains several products that look superficially similar but solve different problems. Understanding the structural lines is the easiest way to evaluate ZCP against them.

**One-shot generators (Lovable, Bolt, v0).** Prompt-to-prototype tools. You describe an app, the agent generates code, you get an artifact you can preview and maybe export. The output exists in isolation — there's no real database behind it, no production path, no persistent state between sessions. ZCP is the opposite end of this spectrum: the agent's output is a running system on production-grade infrastructure from the first deploy. Handoff from agent to human is SSH into the same environment, not exporting a folder and rebuilding it elsewhere.

**Code-execution sandboxes (E2B and similar).** Designed for agents to run code safely in isolation. The sandbox spins up, the agent executes, the sandbox tears down. There's no managed database, no service network, no deploy pipeline — that's by design. ZCP solves the next problem: where does an agent build, deploy, test, and iterate against real infrastructure across hours and sessions. The two solve different layers; they can coexist in a larger system.

**AI-enabled hosted IDEs (Replit Agent and similar).** Replit provides an actual development environment with AI on top, which is closer to ZCP than the generators above. The structural differences are environment fidelity and ecosystem openness. Replit's runtimes and managed services approximate production rather than match it; the IDE, deployment, and hosting are all proprietary. ZCP uses standard Linux, standard Postgres, standard SSH, standard runtimes — nothing about it requires you to stay inside ZCP. Local + VPN lets you connect from any local editor, and the code the agent writes uses the same Node, Python, or Go you'd use anywhere else.

**Agent primitive collections (Cloudflare-style).** This pattern gives agents a set of APIs to call — versioned storage, model gateways, vector search — each a separate product with its own semantics. The agent orchestrates across them. ZCP gives the agent a place to live where everything is on one network, accessible by hostname, no API orchestration required. Different bet on what scales: breadth of primitives versus depth of environment.

**AI-enabled CDEs (Codespaces + Copilot, Gitpod + AI).** Adding AI to a code editor is not the same as giving the agent infrastructure to operate. The agent in those environments can write code; it can't provision a Postgres, deploy a service, or read production logs — those aren't part of the editor's surface. ZCP's MCP server is built around exactly those operations, with permissioning that lets you decide which the agent can perform without asking.

| | Persistent workspace | Real services | Deploy pipeline | Env parity | Standard tooling |
|---|:-:|:-:|:-:|:-:|:-:|
| One-shot generators | – | – | – | – | – |
| Replit | ✓ | ~ | ~ | – | – |
| E2B | – | – | – | – | ✓ |
| Codespaces / Gitpod | ✓ | – | – | – | ✓ |
| Cloudflare-style | – | ~ | ~ | – | – |
